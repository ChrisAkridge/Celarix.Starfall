    - Working from an example slide which is a view of how a single-precision float is built, bit by bit
    - Displayed as ~~254~~ 277 0 bits with a binary point after 128 of them
    - Number is built one bit at a time, starting from its highest set bit over the range down to its least (maximum 24 bits)
    - A bit is shown as a bit with a place value above it in the form 2^k
    - The user would create this entire process with one method on a component they define: SingleConstructionSequence.SequenceFor(float). It would:
        1. Create BinaryTreeContainer as slide root.
        2. Horizontal-split container into parts of heights 1 and 3
        3. In the lower leaf, horizontal-split container into parts of height 1 and 2 (now slide is a stack with heights 1, 1, 2)
        4. In the upper leaf of the lower leaf, add a FlexRowContainer and set the margin to, say, 0.1 container heights.
        5. For each of the 277 bits:
            1. Create a BinaryTreeContainer.
            2. Horizontal-split it into parts of height 1 and 3.
            3. Create a MathTypesettingElement and set font and color. Set id to "bit-k" for the kth bit and add the class "place-value". Set Alignment to Center.
            4. Add a Term with a value "2".
            5. Add a Superscript which has a term with a value equal to that of the exponent.
            6. Add this to the top leaf of the BinaryTreeContainer.
            7. Create a TextElement with a monospace font, white color, and Center alignment. Set the text to "0".
            8. Add this to the bottom leaf of the BinaryTreeContainer.
        6. After the 128th bit, instead of creating a BinaryTreeContainer, create a SingleElementContainer and add a TextElement representing the binary point.
    - This constructs one slide, so let's pause here and see how layout and rendering occurs. This is purely internal to the Helium scene engine.
    - Layout has two phases: sizing and positioning.
    - A slide is a tree of Helium elements, and sizing begins by making a copy of the tree with PositionedHeliumElements that have properties for X, Y, Width, and Height and a reference to the element they're made from.
    - Those four properties are of type Measurement?, which has the following properties:
        - Unit: either ContainerWidth, ContainerHeight, or Pixel
        - Value: 0-1 for ContainerWidth or ContainerHeight, or some whole number of pixels
    - The goal of sizing is that all Width and Height properties on all tree nodes have a pixel value.
    - Each element is asked to set its own size and is handed the size of its parent.
    - Let's see how that works with our slide example:
        1. The slide is known to be, for simplicity, 1000 by 1000 pixels.
        2. The BinaryTreeContainer knows its first-level splits have a 1:3 ratio. It sizes the two splits as 1000x334 and 1000x666.
        3. The second-level splits have a 1:2 ratio. They become 1000x222 and 1000x444, respectively.
        4. The FlexRowContainer is programmed to fill the entire height of its container, so it knows it is 222 pixels tall, but not yet its width. It asks its elements to size themselves.
        5. For each BinaryTreeContainer representing one bit and its place value:
            1. It knows its total height is 222 pixels and that it has 1 split with a 1:3 ratio. Thus, the leaves are 56 and 166 pixels, respectively.
            2. It asks the MathTypeSetting element to size itself, passing the height.
            3. The MathTypeSetting element asks for the size of the first element, a Term, given its height. The Term asks SkiaSharp "if I can be this tall, how wide would I be?" for the text of "2" and gets an answer.
            4. Next, the superscript is considered. It has a default kerning of, say, 0.1 container heights (here, 6 pixels) and an offset from the top of the line of, say, 0.1 container heights (also 6 pixels). It also has, say, a scale factor of 0.5, so it has half the height of the 2. This means it is 28 pixels tall, and it asks SkiaSharp "how wide am I if I'm this tall?" for the superscript.
            5. The width of the 2, the gap to the exponent, and the exponent are summed and reported as the width. Call it 60 pixels, so this place value expression is 60x56.
            6. The same thing is done for the bit. Knowing it has a height of 166 pixels, it asks SkiaSharp how wide the 0 would be, and gets, say, 64 pixels.
            7. The BinaryTreeContainer reports its width as the wider of the two leaves, so it becomes 64x222. All such children report the same. The binary point is also 64x222.
            8. The FlexRowContainer has a margin of, say, 0.05 container heights = 11 pixels, so it adds the widths of all 255 children plus 254 margins and reports its width as 19114 pixels.
        6. We now enter the positioning phase...

1. A rectangular slide is colored dark blue.
2. A row of monospace 0s appears with a consistent gap between each 0. Between 2 of the bits is a white period representing the binary point, which occupies the gap.
3. Above them, gray expressions of the form 2^k appear, representing place values of each bit.

This is enough to start with. Very nice! I forgot to mention how transitions work. So, the PresentationEngine is either rendering a slide OR a transition, only one or the other, never both. A transition is passed the "from" and "to" slides, and while it doesn't need to know exactly every single thing, it is allowed to expect the slides to have a particular structure.

Imagine a property ShowPlaceValues on SingleBitElement. If false, the place values don't show, but nothing else changes - bit spacing is equivalent and they still only take 3/4 of the height of the element. If true, place values render. These two slides with the zero transition just has the place values pop into place.

Let's say I want a transition where the place values slide in from below and fade in. How would I do this? As author of SingleBitElement, I also write ShowPlaceValueTransitionComponent (or maybe just PlaceValueTransitionComponent). What does it do? Let's examine the off-to-on case first. This is PlaceValueTransitionComponent.Render(double progress)

1. Take the "from" slide and clone it.
2. On this clone, find the SingleBitElement with the matching ID (i.e. new PlaceValueTransitionComponent(forId: "my-cool-bits", duration: 1d)) and set ShowPlaceValues to true.
3. Lay the slide out, getting Renderables. Renderables can have IDs made during creation by SingleBitElement, and I've also added HTML-style classes. So the transition next selects the equivalent of #my-cool-bits.place-value and gets all renderables that are place values.
4. It wants them to start from 2 self-heights below where they should end up. They're 75 pixels tall, so that's y += 150 at progress = 0, and they lerp to y += 0 over the progress. Their opacity also lerps from 0 to 1.

That's it! The transition can care about anything, but here, all it cares about is "make yourself renderable so I can move stuff around". The reason it's a transition COMPONENT and not just a transition is that it allows you to make transitions out of multiple components if different stuff has to happen on a single transitions. Like if you have a SingleBitElement but also want to make the background brighten a bit, you can define them separately and add them to a single transition.

How's this design?

Not everything can be represented as property transforms! Transitions definitely are meant to know what slide they come from. I will offer many generic transitions like "just lerp every element with the same ID to wherever it just moved to", but I want people to be able to define transitions for the slides they actually have. "Is renderable present in to but not in from?" I'm going to push back a little on this. The idea of a transition edge between two slide nodes is that it represents a very particular action. This is the "no place values" slide, that is the "yes place values slide", this is the transition between the two. Though it might seem like this cheapens components, I contend it doesn't - it's more than a transition is meant to be something first-class, not like how Manim does it with arbitrary (and IMO ugly) geometry morphs at times. When you press "next slide", you should have a solid way to describe what is happening: "I want to show place values over the bits." "I want to show the target and current values and the arrow pointing at the bit being considered." "I want to..." followed by some verb. This turns the visual language of the presentation into the first-class feature of it.

Absolutely correct. Transitions are not mutations. Once the transition is done, the slide is rendered, and it is the source of truth, so if the transition failed to reproduce the slide, there will be a jarring cut. As for composability, I do agree that a transition boils down to property transforms, and I guess I can make it do that. So if multiple transitions want to use the same element, we get vector addition rather than overriding. The problem is that elements are pretty broad - renderables have position and size, but not every element has opacity or color, and other things transitions want to use. So, instead, I'm going to move it to the transition. A transition (can be) a set of transition components that each take a list of renderables, mutate some properties, and pass it to the next component until they're all done, so we keep that vector addition. That way you can be explicit and say exactly what you want changed.
